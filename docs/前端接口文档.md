## 一、通用规范

### 1.1 请求头

```http
Content-Type: application/json
Authorization: Bearer <token>    // 后续集成认证时使用，当前版本暂不校验
```

### 1.2 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

### 1.3 错误码

| 错误码 | 含义 |
|--------|------|
| `0` | 成功 |
| `40000` | 请求参数错误 |
| `40100` | 未认证（预留） |
| `40300` | 无权限（预留） |
| `40400` | 资源不存在 |
| `50000` | 服务器内部错误 |
| `50100` | LLM 调用失败 |
| `50200` | 向量检索失败 |
| `50300` | 文档处理失败 |

### 1.4 分页响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [...],
    "total": 100,
    "page": 1,
    "page_size": 20
  }
}
```

## 三、对话流式接口（SSE）+ Agent Loop

> 模块：智能对话（Module 2） + 意图识别（Module 3） + RAG 检索（Module 4）+ 思考过程（Module 6） + 引用溯源（Module 5）

### 3.0 Agent Loop（后端内部处理管道）

前端只需要调用一个接口 `/api/chat/stream`，后端内部按以下管道顺序自动执行，无需前端分别调用：

```
用户消息
  │
  ▼
┌─────────────────┐
│  ① 意图识别      │  ← LLM 分类，内置在 Agent Loop 中，无需独立 API
│   (Module 3)    │    结果通过 SSE event: intent_status 推送
└────────┬────────┘
         │ intent = knowledge_qa / general_chat / ...
         ▼
┌─────────────────┐
│  ② 路由分发      │  ← 根据意图决定后续处理链路
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
 knowledge_qa   general_chat
    │              │
    ▼              ▼
┌──────────┐   ┌──────────┐
│③ RAG检索 │   │ 直接生成  │  ← 知识问答走 RAG 管道，闲聊跳过检索
│ Module 4 │   └────┬─────┘
└────┬─────┘        │
     │              │
     ▼              │
┌──────────┐        │
│④ 重排序   │        │
└────┬─────┘        │
     │              │
     ▼              ▼
┌─────────────────────┐
│  ⑤ LLM 生成回答      │  ← token 通过 SSE 流式推送
│   (Module 2)        │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  ⑥ 后处理            │  ← 引用标注、答案校验
│   (Module 5)        │
└─────────────────────┘
```

**意图枚举（后端内部使用，前端通过 SSE 接收结果）：**

| 意图值 | 标签 | 处理链路 |
|--------|------|----------|
| `knowledge_qa` | 知识问答 | RAG 检索 → 重排序 → 生成 → 引用 |
| `general_chat` | 一般对话 | 直接 LLM 生成（无检索） |
| `document_query` | 文档查询 | 检索文档元信息而非内容 |
| `system_command` | 系统指令 | 解析指令并执行对应操作 |

> **前端只需要知道**：发送消息到 `/api/chat/stream`，通过 SSE 事件接收意图结果和生成内容。意图识别、路由分发、RAG 检索等步骤全部在后端 Agent Loop 中自动完成，前端不用关心内部细节。

### 3.1 发送消息（SSE 流式）

```
POST /api/chat/stream
Accept: text/event-stream
```

**请求体：**

```json
{
  "conversation_id": "conv_a1b2c3",
  "message": "变压器巡检有哪些要点？",
  "knowledge_bases": ["kb_power_standard", "kb_device_manual"],
  "params": {
    "top_k": 5,
    "similarity_threshold": 0.7,
    "use_rerank": true,
    "rerank_threshold": 0.5
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `conversation_id` | string | 是 | 会话 ID |
| `message` | string | 是 | 用户输入内容 |
| `knowledge_bases` | string[] | 否 | 指定知识库，空则使用默认配置 |
| `params.top_k` | number | 否 | 检索返回数，默认 5 |
| `params.similarity_threshold` | number | 否 | 相似度阈值，默认 0.7 |
| `params.use_rerank` | boolean | 否 | 是否启用重排序，默认 true |
| `params.rerank_threshold` | number | 否 | 重排序阈值，默认 0.5 |

> 接口会先创建 `user` 消息，流式返回 `assistant` 消息。接口返回结束后，消息已持久化，可通过 2.3 完整获取。

### 3.2 SSE 事件流

详见 [SSE 事件协议详解](#十一sse-事件协议详解)。

```
event: intent_status
data: {"status":"started","label":"正在分析问题..."}

event: intent_status
data: {"status":"done","label":"识别为：知识问答","intent":"knowledge_qa","confidence":0.95}

event: thinking_step
data: {"step":{"type":"retrieval","label":"检索知识库","status":"running"}}

event: thinking_step
data: {"step":{"type":"retrieval","label":"检索知识库","status":"done","detail":"命中 5 条结果"}}

event: token
data: {"text":"根","index":0}

event: token
data: {"text":"据","index":1}

event: citation
data: {"citation":{"id":"1","doc_id":"DOC-001","doc_name":"电力变压器巡检手册.pdf","text":"变压器外壳应保持清洁...","score":0.96}}

event: done
data: {"message_id":"msg_002","total_tokens":256,"latency_ms":3200}

event: error
data: {"code":50100,"message":"LLM 调用超时，正在重试..."}
```

## 四、RAG 检索接口

> 模块：知识增强检索（Module 4）核心接口

### 5.1 语义检索（纯调试，不经过 LLM）

```
POST /api/rag/search
```

**请求体：**

```json
{
  "query": "变压器巡检要点",
  "knowledge_bases": ["kb_power_standard", "kb_device_manual"],
  "top_k": 5,
  "similarity_threshold": 0.7,
  "use_rerank": true,
  "rerank_threshold": 0.5,
  "rerank_top_n": 3
}
```

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `query` | string | 是 | — | 检索查询文本 |
| `knowledge_bases` | string[] | 否 | 全部知识库 | 指定知识库 ID 列表 |
| `top_k` | number | 否 | 5 | 向量检索返回数 |
| `similarity_threshold` | number | 否 | 0.7 | 相似度阈值（0-1） |
| `use_rerank` | boolean | 否 | false | 是否启用重排序 |
| `rerank_threshold` | number | 否 | 0.5 | 重排序分数阈值（0-1） |
| `rerank_top_n` | number | 否 | 3 | 重排序后保留数 |

**响应：**

```json
{
  "code": 0,
  "data": {
    "query": "变压器巡检要点",
    "mode": "vector_rerank",
    "results": [
      {
        "rank": 1,
        "chunk_id": "chunk_abc",
        "doc_id": "DOC-001",
        "doc_name": "电力变压器巡检手册.pdf",
        "text": "变压器外壳应保持清洁，无渗漏油现象。定期检查套管是否清洁完整...",
        "vector_score": 0.92,
        "rerank_score": 0.96,
        "page_number": 12,
        "chunk_index": 5
      },
      {
        "rank": 2,
        "chunk_id": "chunk_def",
        "doc_id": "DOC-002",
        "doc_name": "变压器运行温度标准.pdf",
        "text": "上层油温一般不应超过85°C，温升不超过55°C...",
        "vector_score": 0.88,
        "rerank_score": 0.91,
        "page_number": 3,
        "chunk_index": 2
      }
    ],
    "total_hits": 5,
    "took_ms": 42
  }
}
```

### 5.2 跨知识库联合检索

> 与 5.1 同一接口，`knowledge_bases` 传多个知识库 ID 时自动启用联合检索模式。

**逻辑说明：**

1. 并行检索各知识库（每个库按各自的 `top_k` 检索）
2. 合并结果，按 chunk_id 或文本哈希去重
3. 如果启用重排序：所有结果统一送入 Reranker，按 rerank_score 降序取 `rerank_top_n`
4. 如果未启用重排序：按 vector_score 降序取 `top_k`

### 5.3 检索模式对比测试

```
POST /api/rag/search/compare
```

**请求体：**

```json
{
  "query": "变压器巡检要点",
  "knowledge_bases": ["kb_power_standard"],
  "top_k": 5,
  "threshold": 0.7
}
```

**响应：**

```json
{
  "code": 0,
  "data": {
    "vector_only": {
      "results": [...],
      "took_ms": 18
    },
    "vector_rerank": {
      "results": [...],
      "took_ms": 58
    },
    "comparison": {
      "overlap_count": 3,
      "vector_only_unique": 2,
      "rerank_unique": 0
    }
  }
}
```

## 九、SSE 事件协议详解

### 11.1 通用格式

```
event: <event_type>
data: <json_payload>\n\n
```

### 11.2 事件类型一览

| 事件类型 | 触发时机 | 频率 | 用途 |
|----------|----------|------|------|
| `intent_status` | 意图识别开始/完成 | 1-2 次/请求 | 前端展示意图识别状态 |
| `thinking_step` | 每个思考步骤开始/完成 | 每步骤 2 次 | 前端更新思考步骤列表 |
| `token` | LLM 生成每个 token | 高频 | 前端流式渲染文本 |
| `citation` | 检索到引用时 | 0-N 次 | 前端展示引用标注 |
| `error` | 任何环节出错 | 按需 | 前端展示错误提示 |
| `done` | 回答完成 | 1 次 | 前端清理流式状态 |
| `heartbeat` | 空闲保活 | 每 15s | 防止连接超时断开 |

### 11.3 完整流程序列

```
Client                          Server
  │                                │
  ├── POST /api/chat/stream ──────►│
  │                                │  1. 创建 user 消息 (持久化)
  │                                │  2. 创建 assistant 消息 (空内容)
  │                                │
  │◄── event: intent_status ───────┤  status: "started"
  │◄── event: intent_status ───────┤  status: "done", intent: "knowledge_qa"
  │                                │
  │◄── event: thinking_step ──────┤  type: "retrieval", status: "running"
  │◄── event: thinking_step ──────┤  type: "retrieval", status: "done"
  │◄── event: citation ────────────┤  检索结果引用
  │◄── event: thinking_step ──────┤  type: "generation", status: "running"
  │                                │
  │◄── event: token ──────────────┤  text: "根"
  │◄── event: token ──────────────┤  text: "据"
  │◄── event: token ──────────────┤  text: "知"
  │◄── event: citation ────────────┤  生成中发现的引用
  │◄── event: token ──────────────┤  text: "识"
  │  ... (持续 stream) ...          │
  │◄── event: thinking_step ──────┤  type: "generation", status: "done"
  │◄── event: done ────────────────┤  message_id, tokens, latency
  │                                │  3. 持久化完整 assistant 消息
  │  SSE 连接关闭                   │
```

### 11.4 事件详细结构

#### `intent_status`

```json
// started
{
  "status": "started",
  "label": "正在分析问题..."
}

// done
{
  "status": "done",
  "label": "识别为：知识问答",
  "intent": "knowledge_qa",
  "confidence": 0.95

```

#### `thinking_step`

```json
// running
{
  "step": {
    "type": "retrieval",
    "label": "检索知识库",
    "status": "running"
  }
}

// done
{
  "step": {
    "type": "retrieval",
    "label": "检索知识库",
    "status": "done",
    "detail": "命中 5 条结果，耗时 42ms"
  }

```

**步骤 type 枚举：**

| type | 标签 | 顺序 |
|------|------|------|
| `intent` | 识别意图 | 1 |
| `retrieval` | 检索知识库 | 2 |
| `generation` | 生成回答 | 3 |
| `verify` | 验证答案 | 4（可选） |

#### `token`

```json
{
  "text": "据",
  "index": 1,
  "logprob": null
}
```

#### `citation`

```json
{
  "citation": {
    "id": "1",
    "doc_id": "DOC-001",
    "doc_name": "电力变压器巡检手册.pdf",
    "chunk_id": "chunk_abc",
    "text": "变压器外壳应保持清洁，无渗漏油现象...",

  }

```

> 引用 `id` 为从 1 递增的序号，用于在答案文本中标注 `[1]`、`[2]`。

#### `done`

```json
{
  "message_id": "msg_002",
  "total_tokens": 256,
  "prompt_tokens": 1200,
  "completion_tokens": 256,
  "latency_ms": 3200
}
```

#### `error`

```json
{
  "code": 50100,
  "message": "LLM 调用超时",
  "fatal": false
}
```

| 字段 | 说明 |
|------|------|
| `fatal: true` | 无法恢复，SSE 流将关闭 |
| `fatal: false` | 可恢复（如重试中），流继续 |

#### `heartbeat`

```json
{}
```

> 每 15 秒发送一次，防止代理/网关因空闲断开连接。

### 11.5 前端 SSE 接收实现参考

```typescript
// 使用 fetch + ReadableStream（推荐，比 EventSource 更灵活，支持 POST）
async function streamChat(params: ChatRequest): Promise<void> {
  const response = await fetch('/api/chat/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(params),
  });

  const reader = response.body!.getReader();
  const decoder = new TextDecoder();
  let buffer = '';

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop() || '';

    let eventType = '';
    for (const line of lines) {
      if (line.startsWith('event: ')) {
        eventType = line.slice(7).trim();
      } else if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        handleSSEEvent(eventType, data);  // 分发到对应处理器
      }
    }
  }
}
```
